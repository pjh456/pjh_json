# SIMD 优化细节

本文说明 `pjh_json` 在解析过程中的 SIMD 使用方式、64 字节 padding 的意义，
以及为何热路径会被设计成可预测的分支结构。

## 为什么使用 SIMD
JSON 中有大量可快速跳过的字符区间，最消耗性能的两类场景是：
- 跳过空白
- 字符串扫描（寻找 `"` 与 `\`）

SIMD 允许一次检查 16/32/64 字节，能把分支密集的循环变成并行比较。

## Padding 策略
解析器要求输入 buffer 至少追加 64 字节 `\0`，通过以下接口保证：
- `parse_copy`：复制输入并追加 64 字节 padding
- `parse_in_situ`：要求调用方提供已 padding 的可写 buffer
- `parse_file`：读文件到 buffer 并追加 padding

好处：
- SIMD 读取可以越过逻辑末尾而不越界
- `\0` 可作为“安全终止符”使用
- 省掉部分 `m_curr < m_end` 的检查

## 空白跳过（`Parser::skip_whitespace`）
关键观察：JSON 结构字符、数字、字母都大于 `0x20`，
只有 4 个空白字符小于等于 `0x20`（空格、`\t`、`\r`、`\n`）。

热路径：
- 若 `*m_curr > 0x20`，立即返回，不走 SIMD。

SIMD 路径：
- 读取一个 SIMD batch
- 计算 `is_ws = (b <= 0x20) & (b != 0)`
- 生成 mask 并反转，用于定位第一个非空白或 padding
- `std::countr_zero` 快速跳到目标位置

速度来源：
- 循环体几乎无不可预测分支
- `b != 0` 使 `\0` padding 触发停下

## 字符串扫描（`Parser::parse_string`）
字符串解析要发现三类字符：
- 终止引号 `"`
- 转义字符 `\`（慢路径）
- 非法控制字符 `< 0x20`

SIMD 路径：
- 并行比较 `"`、`\` 与 `< 0x20`
- 组合 mask，定位第一个特殊字符

常见情况偏置：
- 大多数字符串没有转义
- 可连续跳过大量普通字符

慢路径触发：
- 遇到 `\`，进入原地重写循环
- 遇到 `< 0x20`，抛出解析错误

## 字面量解析（`Parser::parse_literal`）
`true/false/null` 通过 4/8 字节对比完成：
- `std::bit_cast` 构造机器端序常量
- `std::memcpy` 读取 4/8 字节避免别名问题
- `false` 使用掩码忽略 padding 影响

这样能把多次字符比较压缩为一次寄存器比较。

## 设计权衡
SIMD 带来如下约束：
- 需要连续且 padding 的输入
- 部分逻辑更复杂
- 依赖 `xsimd`

这些复杂度换来的收益是：在不牺牲正确性的前提下，
大幅降低 JSON 解析的平均成本。
